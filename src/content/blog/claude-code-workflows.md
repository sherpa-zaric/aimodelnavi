皆さんこんにちは、刘小排です。

先週、Claude Codeの新バージョンがOpus 4.8と同時にリリースされ、新しい機能 /workflows が追加されました。

数日前、/workflows の使用場面について皆様の意見を募集したところ、刘润さんからいくつかのスクリーンショットを共有していただきました。

潤さんが Claude Code の /deep-research（実質的には /workflows の一種）を使って調査報告書を作成しているのを発見しました。効果は非常に良く、100以上のエージェントを同時に起動し、役割分担が明確で、秩序井然としていました。

これは私にいくつかのインスピレーションを与えました：**以前はAIエージェントがこなすのがやや厳しかった場面で、/workflows を使うことで理論的に完成品質が向上できるケースが存在するだろうか？**

もし、そんな場面があるなら、以下の条件を満たすはずです：

- 1Mのコンテキストウィンドウでは足りず、/workflows を使うことで「実際にいくつのエージェントが必要か」を自動的に計画し、「容量不足」の問題と「複数エージェントの分担を面倒くさがる」問題の両方を解決できる。
- タスクが大規模に分割可能で、並列処理に変換できる。

2日間の実験をした結果、Claude Code 20x Max の週間制限枠を2日で使い切りました（はい、/workflows はトークン消費が非常に激しいですが、幸いなことに今週の週間制限枠がリセットされました）。

そろそろ皆さんと共有するタイミングです。

さて、この2日間で何を試したのか？

まず、最も気持ちよかったことを紹介します。

Raphael のコードベースをスキャンして、「低リスクで安心して修正できる」古いバグを探させました。

あの手のバグを知っているだろうか――**致命的ではなく、緊急でもないが、確かに存在していると分かっているもの。**

そのコードを通るたびに不安になるが、わざわざ午後を使って修正することはない。なぜなら、もし一つを修正して三つが壊れたら？と考えるからだ。

そうして、バグは「古びたまま」放置される。一年、二年。

以前は私もそのままにしていました。しかし今回は、/workflows を一投すると、コードベースを12のセクションに自動分割し、31のエージェントが同時にスキャン。一批は検索し、一批は細部をチェックし、確信度が低いものは即座に除外しました。

2時間後、リストが渡されました。

見直してみると――へえ、これだ！まさに「明らかに修正すべきだが、誰も手をつけていなかった」古いバグの集まりでした。
![画像](/images/blog/claude-code-workflows/img-1.png)
その時の感想は「うわ、こんなやつら、ようやく誰かが見つけてくれたか」というものでした。

次に、全く違う作業を試しました：「この難題について、他人は通常どう解決するか」。

これは潤さんのインスピレーションからです。

/workflows だけ使う場合：情報のエコチェンバーになりかねない。
/deep-research だけ使う場合（以前のClaudeウェブ版にもあった）：現在のコードベースを読むことができない。

そこで、私の方法は：**难题に直面したら、/workflows と /deep-research を同時に使う**です。

下図のように、一文で105のエージェントが起動しました。
**最終効果は非常に、非常に素晴らしい！ぜひ試してみてほしい！**
![画像](/images/blog/claude-code-workflows/img-2.png)
試してからもさらに考えました。/workflows は一体何をもたらすのだろう？

考えて考えた末、一言で言えば――

**多くの場合、AIの知能レベルはとっくにオーバースペックで、ボトルネックは体力不足にある。**

どんなに賢い人でも、コードベース全体を読みきれない――頭が悪いのではなく、容量が足りないのだ。

どんなに有能な人でも、50のファイルを同時に編集できない――手が遅いのではなく、手が一組しかないのだ。

/workflows は Claude をより賢くするのではなく、Claue をより「多」くする。

したがって、ある作業が /workflows に適しているかどうかは、自分に一句問い質せばよい：**これは「知能の問題」か、それとも「体力の問題」か？**

ひらめきが必要で、一貫した判断力を要するもの――それは知能の問題だ。

「一人の賢者が面倒くさがり、五十の普通人が分担すればできる」ようなもの――それは体力の問題で、 such a case、/workflows を放り込み、さらに /deep-research を重ねれば、ダブルバフとなり、速くてよい結果が出る。

これがAIエージェントツールの新たな変化の潮流かもしれない。

すでに天才的なAIをさらに天才にするのではなく、AIが苦労する作業により適するようにすることだ。

もし記事が長すぎると感じたら、蒸留版を提供しよう：最初に覚えておくべきことはこれだ。

> 
**Claude Code を使って真正面から难题にぶつかる際、通常の問題記述の後に /workflows /deep-research を追加するだけで、効果が驚異的になる**

CodexにDeepSeek、GLM、Kimiのような第三者の大規模モデルAPIキーを直接入力しても、期待通りには動作しないかもしれない。

問題はキー自体やアカウントにあるわけではない。プロトコルが完全に一致していないため、正常な通信ができないのだ。

現在の新版Codex CLI / Codex AppのAPIキーモードでは、デフォルトの通信路は**OpenAI Responses API**となっている。しかし、多くの第三者モデルが提供するのは**OpenAI互換のChat Completions**である。

メッセージ構造、ストリーミングレスポンス、推論フィールド、ツール呼び出しの表現方法などに差異があるため、「ベースURLとキーを変更するだけで直接動く」とは単純に考えられない。

![Codexが第三者モデルに接続する際のプロトコル変換関係](/images/blog/codex-deepseekv4-glm51-k26/img-1.webp)

そのため、第三者モデルを接続する上で本当に解決すべきは「APIキーを入力すること」ではなく、以下のような点だ：

**Codexが送信するリクエストを第三者モデルが理解できるようにし、第三者モデルが返す内容をCodexが認識できる形に変換すること。**

現在知られている一般的な方法は2つある：

- **CC-Switch**：ローカルプロキシとプロトコル変換を利用する。
- **Codex++**：Codexデスクトップ版の拡張と設定注入に重点を置く。

どちらもCodexに第三者モデルを接続するのに役立つが、問題解決の場所が異なる。前者は主にプロキシ層でプロトコルを変換し、後者はCodexデスクトップ版の設定やUI層で拡張を行う。

第三者モデルの接続だけが目的ならCC-Switchが優先される。Codexデスクトップ版を使用し、プラグインエントリやUI拡張も望む場合はCodex++を検討する。

## 1. CC-Switchルート：低侵入の設定切替方案

### 1.1 CC-Switchとは

CC-Switchは、複数のAIコーディングツールの設定センターおよびローカルルーティングプロキシのようなものだ。当初はClaude Code用に作られ、後にCodex、Gemini CLI、OpenCode、OpenClawなどのツールに拡張された。

このオープンソースプロジェクトは、AI時代に作られた優秀な製品の一つであり、AIの波に乗って人気を博し、現在のスター数は9万に近づいている。疑問の声もあるが、作者が真剣にメンテナンスしており、体験も良好であることは否めない。

![画像](/images/blog/codex-deepseekv4-glm51-k26/img-2.png)

Codexの場面では、主に2つのことを実行する：

- **設定切替**：異なるコーディングツールの設定を統合管理し、プリセットへのワンクリック切替、テンプレートのインポート、サプライヤーの切替をサポートする。
- **ローカルプロキシ**：ローカルマシンでHTTPサービスを起動し、Codexのリクエストに対してプロトコル変換とルーティング配信を行い、第三者モデルに転送する。

核となるポイントは、Codex自体を変更せずに設定を変更し、プロキシを起動することだ。

### 1.2 インストールと設定

設定を行う前に、Codexが現在APIキー/ローカル設定ルートを使用しており、ChatGPTログインルートではないことを確認することを推奨する。2つのモードが混在すると、リクエストパスが不明確になり、エラーの判断が難しくなる。また、Codexを少なくとも1回実行し、設定ファイルを初期化しておくと、後続のルーティング設定が容易になる。

1. CC-SwitchのGitHubリポジトリ[1]を開き、インストーラーをダウンロードしてインストールする。
2. CC-Switchを起動し、右上の`+`をクリックしてサプライヤーを追加する。ここではDeepSeekを例に挙げる。

![画像](/images/blog/codex-deepseekv4-glm51-k26/img-3.png)
![CC-Switchによるサプライヤーの追加](/images/blog/codex-deepseekv4-glm51-k26/img-4.png)
3. 一般的にはAPIキーを入力するだけで良い。ローカルルーティングマッピングが必要な場合は、`Needs Local Routing`が有効になっていることを確認する。通常、デフォルトで有効になっている。

![CC-SwitchでのAPIキー入力](/images/blog/codex-deepseekv4-glm51-k26/img-5.png)
![CC-Switchのローカルルーティング設定](/images/blog/codex-deepseekv4-glm51-k26/img-6.png)
4. メインページに戻り、左上の歯車アイコンをクリックして設定に入り、「ルーティング」を開く。下図のように、ローカルルーティングボタンをオンにする。この時、下に表示される総リクエスト数が`0`であれば、後でCodexが実際にCC-Switchを経由しているかどうかを判断できる。

> CC-Switchの設定画面。左上の歯車アイコンをクリックし、「ルーティング」セクションを開いた状態。ローカルルーティングのオン/オフボタンが表示されている。
![CC-Switchでローカルルーティングを有効にする](/images/blog/codex-deepseekv4-glm51-k26/img-8.png)
5. Codexを再起動し、メッセージを送信して、正常に返信があるか確認する。

![Codexが第三者モデルを使用して返信する様子](/images/blog/codex-deepseekv4-glm51-k26/img-9.png)
6. CC-Switchのルーティング設定画面に戻り、リクエスト履歴が表示されれば、設定が有効になったことが分かる。

![CC-Switchのリクエスト履歴](/images/blog/codex-deepseekv4-glm51-k26/img-10.png)

以上で、CC-Switchルートは動作する。

## 2. Codex++ルート：より深い拡張方案

### 2.1 Codex++とは

Codex++は汎用的なプロトコル変換プロキシではなく、Codexデスクトップ版の外部拡張ランチャーのようなものだ。

BigPizzaV3/CodexPlusPlus[2]を例に挙げると、Codex Appのオリジナルインストールファイルを変更するのではなく、外部ランチャーを通じてCodexを起動し、Chromium DevTools Protocol（CDP）を使用してCodexのレンダリングプロセスに拡張スクリプトを注入する。サプライヤーの設定は、独立した管理ツールで`~/.codex/config.toml`に書き込まれる。

![画像](/images/blog/codex-deepseekv4-glm51-k26/img-11.png)

そのため、CC-Switchとは重点が異なる：

- CC-Switchは主にリクエストの転送方法とプロトコル変換を解決する。
- Codex++はデスクトップ版の拡張方法、設定の注入方法、プラグインエントリの使用方法により注目する。

ここで混同しやすい点に注意が必要だ。別の独立プロジェクトb-nnett/codex-plusplus[3]もあり、それは`app.asar`を変更してLoaderを注入する方式で、本文で説明するBigPizzaV3版とはアーキテクチャが全く異なる。以下で述べるCodex++は、特に断りのない限りBigPizzaV3版を指す。

### 2.2 インストールと設定

1. BigPizzaV3/CodexPlusPlus[4]を開き、システムに対応するインストーラーをダウンロードする。
2. インストール後、デスクトップに`Codex++`と`Codex++管理ツール`という2つのエントリが現れる。管理ツールはサプライヤーの設定に使用し、`Codex++`は元のCodexアイコンの代わりに起動するために使用する。

`Codex++`ランチャーは、CDPを通じてCodexに拡張スクリプトを注入するために、先に実行する必要がある。元のCodexアイコンを直接クリックすると、このステップがスキップされ、拡張機能が有効にならない。

![Codex++インストール後のエントリ](/images/blog/codex-deepseekv4-glm51-k26/img-12.png)

3. `Codex++管理ツール`を起動し、サプライヤー設定を選択して、サプライヤーを追加する。DeepSeekを例に挙げると：
   - 接続モード：「純API」を選択する。
   - Base URL：Codex++の内蔵プリセットまたはインターフェースの指示に従って入力する。DeepSeek公式ドキュメントが現在提供するOpenAI互換のBase URLは`https://api.deepseek.com`である。ツールがOpenAIスタイルの`/v1`を要求する場合は、`https://api.deepseek.com/v1`と入力する。
   - API Key：自分のキーを入力する。
   - アップストリームプロトコル：`Chat Completions`を選択する。

![Codex++でのサプライヤー追加](/images/blog/codex-deepseekv4-glm51-k26/img-13.png)
![Codex++でのDeepSeek設定](/images/blog/codex-deepseekv4-glm51-k26/img-14.png)
4. 設定完了後、`Codex++`アイコンからCodexを起動する。Codexがすでに実行中の場合は、一度終了してから起動するか、右上の再起動ボタンをクリックする。元のCodexデスクトップアイコンを直接開くと、拡張スクリプトが介入せず、ネイティブのCodexが実行される。

5. 最終的なページ効果は以下の通り：

![Codex++のページ効果](/images/blog/codex-deepseekv4-glm51-k26/img-15.png)

### 2.3 Codex++が何をするのか

Codex++の拡張スクリプトは主に3つのことを行う。

第一に、**第三者設定の書き込み**である。すべてのリクエストを傍受して変更するのではなく、カスタムプロバイダーをCodexのネイティブ設定（例：`~/.codex/config.toml`）に書き込むことで、Codex自体がそのプロバイダーを通じて第三者サービスにアクセスできるようにする。簡単に言えば、それは「Codexにルートを設定してあげる」ようなものであり、ネットワーク層でリクエストを強制的に横取りするわけではない。

第二に、**Codex Appへのエントリ追加**である。Codex++からCodexを起動すると、CDPを使用して拡張スクリプトが注入され、上部メニューバーにCodex++の状態と設定エントリが追加される。実際にサプライヤーの追加や設定の切替を行うのは、独立した「Codex++管理ツール」で行われ、Codexの元のウィンドウ内で直接設定されるわけではない。

第三に、**デスクトップ版の拡張能力の補完**である。例えば、APIキーモードでは、CodexのネイティブプラグインエントリがChatGPTへのログインを要求する場合がある。Codex++で起動すると、このエントリのロックが解除され、セッション削除、Markdownエクスポート、Timeline、プロバイダー同期などの拡張機能が追加される。

![画像](/images/blog/codex-deepseekv4-glm51-k26/img-16.webp)

## 3. 接続原理：Codexは実際何を接続しているのか

上記の2つのルートは一見異なるが、実際に詰まるポイントは同じだ。Codexが送信するモデルリクエストを、第三者サービスが受け止められるかどうかである。

APIキーモードを例にすると、Codexの動作は大まかに以下の通りだ：

```
ユーザー入力
-> Codexエージェントがタスクをオーケストレーション
-> ~/.codex/config.tomlを読み込む
-> model_providerに基づいて対応するプロバイダーを見つける
-> base_url、APIキー、wire_apiなどの設定を取り出す
-> モデルサービスにリクエストを送信
-> モデルが結果を返す
-> Codexがレスポンスを解析
-> ツール呼び出し、ファイル編集、コマンド実行などのアクションを続行
```

ここで最も重要なのは以下の設定項目だ：

- `model_provider`：現在使用するモデルサプライヤー。
- `base_url`：リクエストの送信先。OpenAI、第三者の中継、ローカルプロキシ、または会社内部のモデルゲートウェイなど。
- `env_key`：APIキーをどの環境変数から読み取るか。設定ファイルに直接キーを記述することを避ける。
- `wire_api`：Codexがモデルサービスとの通信に使用するプロトコル。例えば`responses`または`chat`。

最後の`wire_api`が特に重要だ。

一般的なチャットインターフェースが1文を返せるからといって、Codexのエージェントフローをサポートできるわけではない。Codexは単にモデルに質問を送って答えを表示するだけでなく、ストリーミングレスポンス、ツール呼び出し、推論、状態フィールドを解析し、その後もファイルの読み取り、コードの編集、コマンドの実行を続ける必要がある。

因此、第三者モデルを接続できるかどうかは、「OpenAI互換かどうか」だけでなく、それがChat Completionsに互換性があるか、またはCodexが現在使用するResponses通信路を完全に受け止められるかにも依存する。

### 3.1 なぜベースURLだけを変更できないのか

Codexは現在、OpenAI Responses APIの使用を好むが、多くの第三者モデルが提供するのはChat Completions APIだ。两者は同じものではない。

Responses APIはエージェントの場面により適しており、更多的な状態とイベント構造に関与する。Chat Completions APIは従来の対話インターフェースに近く、核心は`messages`リストとモデルの返信である。通常のチャットツールでは、`messages`を送信できれば十分だが、Codexはツール呼び出し、ストリーミングイベント、コンテキスト、推論、タスク状態などの内容を処理する必要がある。

因此、第三者モデルをCodexに接続する際の実際の難点は「リクエストを送信できない」ことではなく、「リクエストとレスポンスが双方で正しく理解できるかどうか」にある。

### 3.2 CC-Switchの原理：プロキシ層での変換

CC-Switchのローカルプロキシが行うのはプロトコル変換である：

- Codexから送信されるResponsesリクエストをChat Completionsに変換し、アップストリームに転送する。
- アップストリームから返されるSSEストリーミングレスポンスを再びResponsesとして再包装し、Codexにプッシュする。
- 推論コンテンツ、ツール呼び出し、`previous_response_id`などの状態フィールドを処理する。

也正是この変換層があるため、第三者モデルが安定して動作するかどうかは、モデル自体だけでなく、プロバイダーの互換性とプロキシの実装にも依存する。

アップストリームがResponses APIをサポートしている場合、プロキシはChat Completionsへの変換層を省き、認証注入、使用量追跡、ヘルスチェックなどの作業を主に担うことができる。

![CC-Switchのプロトコル変換原理](/images/blog/codex-deepseekv4-glm51-k26/img-17.webp)

### 3.3 Codex++の原理：デスクトップ版での拡張

Codex++は**Codex Appデスクトップ版の拡張とプロバイダー設定の書き込み/同期**により重点を置く。CC-Switchのようにすべてのリクエストをローカルプロキシ層に統合してプロトコル変換を行うのではなく、ランチャーを通じてCodexを起動し、CDPを使用して拡張スクリプトを注入することで、Codex Appに追加のメニューや設定エントリ、プラグインエントリ、サプライヤー切替機能を持たせる。

簡單に言えば、**CC-Switchは主に「リクエストのルーティング方法とプロトコル変換方法」を解決し、Codex++は主に「Codexデスクトップ版の拡張方法、および第三者プロバイダーのより便利な書き込みと切替方法」を解決する。**

## 4. 選択ガイド：CC-SwitchかCodex++か

結論から言うと、ほとんどのユーザーはCC-Switchを選択すれば良い。これが私の推奨するデフォルトルートでもある。

### 4.1 ニーズに基づく選択

- 主にCodex CLIを使用し、同時にClaude Code / Gemini CLIも実行する場合：CC-Switchを選択する。
- Codexデスクトップ版のみを使用し、プラグインエントリやUI拡張も望む場合：Codex++を選択する。
- Codexのインストールファイルを一切変更したくない場合：CC-Switchを優先する。
- プロトコル変換とローカルルーティングがすぐに使えることを望む場合：CC-Switchを優先する。
- デスクトップ版の拡張、プラグインエントリ、スクリプト注入をいじりたい場合：Codex++を検討する。

![CC-SwitchとCodex++の選択提案](/images/blog/codex-deepseekv4-glm51-k26/img-18.webp)

### 4.2 機能互換性

第三者モデルを接続した後、すべてのCodex機能が完全に代替できると期待すべきではない。現実的な状況は以下の通りだ：

**完全に使用不可または同等の代替が難しいもの：**

- **Image Gen**：OpenAIの対応する画像生成能力に依存しており、第三者のテキストモデルでは直接代替できない。
- **Computer Use**：Responses APIに内蔵されたcomputer actionタイプ、ローカルランタイム、スクリーンショットフィードバックループに依存している。Chat Completionsプロトコルと通常の第三者モデルは同等の能力を提供できず、プロトコル変換層でも補うのが難しい。

**グレードダウン可能：**

- **普通のSkills / プラグイン**：Codex++のページ拡張と組み合わせることで、一部の場面で使用可能。安定性はバージョンによる。
- **ツール呼び出し**：基本的なコード編集、ファイル読み書き、コマンド実行は通常実行できるが、複雑なツール呼び出しや長時間タスクでは、フォーマットや互換性の問題が生じる可能性がある。

**基本正常：**

- コード作成
- デバッグとリファクタリング
- ファイル読み書き
- プロジェクト管理
- マルチターン対話
- タスクプランニング

### 4.3 より合理的な使用方法

- 軽量なコード質問、テキストタスク、シンプルなスクリプト：DeepSeekなどのコストが低いモデルには、価格面で明らかな優位性がある。
- 複雑なエンジニアリングプロジェクト：まずより強力なGPTでプランニングを通し、その後第三者モデルで単純なサブタスクを処理する。
- GPT Plus / Proの十分な割り当てがある場合：ネイティブ体験が依然として最も安定しており、追加で調整する必要はないかもしれない。
